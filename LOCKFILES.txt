Matey Lockfile + Checkpoint Proposal (v1, per-step checkpoints required)

Status
- Canonical proposal for lockfile-driven schema operations.
- Replaces the prior fast/strict split with one default mode plus `--clean`.

Goals
- Deterministic schema validate/diff/regen with explicit integrity checks.
- Fast divergence detection between base and head migration history.
- Strong tamper detection for migration and checkpoint artifacts.
- Keep artifacts reviewable in git (plain SQL + TOML).

Non-Goals
- No best-effort fallback to unchecked artifacts.
- No optional checkpoint policy in v1.

Repository Layout
1. Single-target
- `db/schema.sql`
- `db/migrations/*.sql`
- `db/checkpoints/*.sql`
- `db/schema.lock.toml`

2. Multi-target
- `db/<target>/schema.sql`
- `db/<target>/migrations/*.sql`
- `db/<target>/checkpoints/*.sql`
- `db/<target>/schema.lock.toml`

Path rule
- Paths stored in lockfile are relative to the lockfile directory.
- No `db/` prefix inside lock entries.

Lockfile Shape (`schema.lock.toml`)
- One `[[step]]` per migration.
- Each step MUST include the checkpoint and schema digest after that step.

Example
lock_version = 1
hash_algorithm = "blake2b-256"
canonicalizer = "matey-sql-v1"
engine = "postgres"
schema_file = "schema.sql"
head_index = 2
head_chain_hash = "..."
head_schema_digest = "..."

[[step]]
index = 1
version = "202602260101"
migration_file = "migrations/202602260101_create_widgets.sql"
migration_digest = "..."
chain_hash = "..."
checkpoint_file = "checkpoints/202602260101.sql"
checkpoint_digest = "..."
schema_digest = "..."

[[step]]
index = 2
version = "202602260102"
migration_file = "migrations/202602260102_add_source.sql"
migration_digest = "..."
chain_hash = "..."
checkpoint_file = "checkpoints/202602260102.sql"
checkpoint_digest = "..."
schema_digest = "..."

Required invariants
- `index` is contiguous and starts at 1.
- Exactly one step per migration version.
- Every step has a checkpoint.
- `head_index == len(step)`.
- Last step `chain_hash == head_chain_hash`.
- Last step `schema_digest == head_schema_digest`.
- Zero-step case is valid:
  - `head_index = 0`
  - no `[[step]]` entries
  - `head_chain_hash = chain_0`
  - `head_schema_digest` is the canonical digest of empty schema state.

Hashing + Canonicalization
1. File digest
- `blake2b(file_bytes, digest_size=32).hexdigest()`.

2. Chain seed
- `chain_0 = blake2b("matey-lock-v1|<engine>|<target>")`.

3. Chain step
- `chain_i = blake2b(chain_{i-1} + "|" + version + "|" + migration_file + "|" + migration_digest)`.

4. Schema digest
- Digest of canonicalized SQL output (canonicalizer versioned by `canonicalizer`).

A/B Definitions
- A: current workspace `schema.sql` canonicalized.
- B: computed schema from replay anchored on verified base checkpoint plus head migration tail.

A invariant
- Before A-vs-B compare, A digest MUST equal `head_schema_digest` in head lockfile.

Base and Head Input Rules
- Base side: merge-base snapshot (or `--base` override), always committed content.
- Head side: current workspace content.
- Same algorithm in local and CI.

Default Algorithm (validate, diff, regen)
1. Resolve base ref and obtain base snapshot.
2. Load base lockfile and head lockfile.
3. Verify both lockfiles fully:
- lock schema validity
- step continuity/order
- migration/checkpoint file existence
- digests for migration/checkpoint files
- chain progression
- head fields match last step
4. Find first divergence index between base and head step chains.
   - If no divergence is found across shared prefix and lengths are equal:
     - define `divergence = head_index + 1` (no-divergence case).
   - If shared prefix matches but one side has extra steps:
     - `divergence = shared_prefix_len + 1`.
5. Build A and verify A digest equals head lock digest.
6. Build B:
- if divergence is first step, anchor is empty scratch.
- else anchor is base step `divergence - 1` checkpoint.
- load anchor checkpoint into scratch.
- replay head migrations from `divergence .. head_index` (no replay when `divergence = head_index + 1`).
- dump canonical schema as B.

Checkpoint Replay Execution Contract
- Anchor load uses `dbmate load` on the selected checkpoint SQL.
- After anchor load, replay MUST run only tail migrations (`divergence .. head_index`), not the full migrations directory.
- Tail replay should use a temporary migrations directory containing only tail files (ordered).
- Do not rely on `schema_migrations` side effects from `dbmate load` (engine-agnostic behavior is not guaranteed).
- Running full `dbmate up` after checkpoint load is invalid and can reapply already-materialized schema objects.
7. Command action:
- `schema validate`: fail unless `A == B`.
- `schema diff`: print diff `A vs B`, non-zero on differences.
- `schema regen`: write B to `schema.sql`, regenerate all step digests/checkpoint digests/head fields deterministically.

`--clean` Semantics
- Ignore checkpoint anchoring.
- Replay full head migration chain from empty scratch.
- Used for strict integrity checks and recovery when no usable anchor exists.
- Scope:
  - supported on `schema validate` and `schema regen` only.
  - not supported on `schema diff`.
  - not supported on `db diff`.

Historical Rewrite Handling
- Rewrite is detected by chain divergence.
- Default mode remains valid if a pre-divergence base checkpoint exists.
- If no usable anchor exists, command fails with explicit instruction to run `--clean`.

Command Surface (v1, normalized namespaces)
- `schema` namespace is artifact-only (repo files + scratch), never live DB.
- `db` namespace is live-DB-only.
- `lock` namespace is lockfile maintenance/checks.

Artifact commands
- `schema validate`
- `schema diff`
- `schema regen`

Live commands
- Canonical mutators: `db up`, `db migrate`, `db down [N]`
- Read/ops helpers: `db status`, `db dump`, `db load`, `db create`, `db drop`, `db wait`, `db diff`
- No rollback alias: `db rollback` is removed from the v1 surface.
- Mutator semantics:
  - `db up`: create-if-needed + apply pending migrations.
  - `db migrate`: apply pending migrations only; fail if DB is missing/unreachable.
  - `db down [N]`: roll back `N` steps (default 1).

Lock commands
- `lock doctor`
- `lock sync`

Top-level DB aliases
- Remove top-level aliases (`matey up`, `matey rollback`, etc.).
- Canonical interface is `matey db ...` only.

No separate lockfile workflow is required in normal operation; lock/checkpoint metadata is managed by `schema regen` (with optional explicit `lock sync`).

Command Extensions (v1)
1. `schema validate --down`
- Default `schema validate` remains forward-only.
- `--down` enables downgrade round-trip validation for changed tail (`divergence .. head_index`).
- Behavior:
  - run normal forward A/B validation first.
  - determine changed tail from divergence.
  - fail if any changed migration has no `down` (missing `down` is irreversible in v1).
  - in scratch, apply base anchor + changed tail to head.
  - roll back exactly changed-tail steps to base.
  - dump canonical schema and require equality with base anchor schema digest.
- Scope: schema reversibility only; does not prove data preservation.

2. `lock doctor`
- Read-only lock health check (no replay, no scratch DB).
- Verifies:
  - lock format/version/canonicalizer.
  - step ordering/index continuity.
  - migration/checkpoint existence.
  - migration/checkpoint digests.
  - chain progression + head fields.
  - `schema.sql` digest equals `head_schema_digest`.
- Exit `0` healthy; non-zero on any failure.

3. `lock sync`
- Deterministically rewrites `schema.lock.toml` from current repo artifacts.
- Recomputes:
  - per-step migration/checkpoint/schema digests.
  - per-step chain hashes.
  - `head_index`, `head_chain_hash`, `head_schema_digest`.
- Does not run DB operations.
- Does not generate checkpoints; missing checkpoint files are an error.

4. `db up` / `db migrate` / `db down` live mutation safety contract
- `db up`, `db migrate`, and `db down` are guarded commands, not raw passthrough.
- Precheck (before mutation):
  - run `lock doctor` (or equivalent internal checks).
  - dump live schema to temp and compare against expected schema at live-applied version.
  - fail closed on drift unless explicit override (`--force`).
- Mutation:
  - execute dbmate mutator with `--no-dump-schema` (repo `schema.sql` must never be rewritten by live commands).
- Postcheck (after mutation):
  - dump live schema to temp.
  - compare to expected target schema:
    - for `db up` / `db migrate`: expected head schema.
    - for `db down [N]`: expected schema at `head-N` target.
  - fail non-zero on mismatch.
- `db diff` remains available as standalone drift check and reuses the same expected-schema semantics.

5. `--down` scope
- Supported on `schema validate` only.
- Not added to `schema diff`.
- Not added to `schema regen`.
- Rationale: `diff`/`regen` are forward artifact operations; down-check is a validation concern.

6. Split `schema diff --live` into `schema diff` and `db diff`
- `schema diff` is artifact-only and never accepts `--live`.
- `db diff` compares live DB schema vs lock-verified expected schema.

Subflag mapping
- `schema validate` supports:
  - `--clean`
  - `--down`
- `schema regen` supports:
  - `--clean`
- `schema diff` supports:
- `db diff` supports:
  - live URL selection via target config / global `--url` override
- `schema diff` and `db diff` do not accept:
  - `--clean`
  - `--expected`
  - `--live`
- Scratch retention uses global `--keep-scratch` only (no command-local duplicates).

CLI De-duplication Policy (v1)
1. Keep one way to do each action.
- If a dedicated subcommand exists, remove duplicate flag forms.
- If a global flag exists, remove duplicate local flag forms.

1a. Canonical DB mutator verbs.
- Canonical mutators are `db up`, `db migrate`, and `db down [N]`.
- `db rollback [N]` is removed from the v1 surface.
- `db up` and `db migrate` are both canonical with distinct semantics.

2. Remove duplicate `--keep-scratch` flags.
- Keep global `--keep-scratch` only.
- Remove command-local `--keep-scratch` from `schema validate`, `schema regen`, and `schema diff`.

3. Remove duplicate template print paths.
- Keep `ci print` and `cd print`.
- Remove `ci init --print` and `cd init --print`.
- `ci init` / `cd init` become write-only commands with `--force`.

4. Keep URL override semantics single-surface.
- Keep global `--url` and `--test-url`.
- `schema diff` does not consume live URL behavior (moved to `db diff`).

5. Remove legacy schema-mode flags from lockfile workflow surface.
- Remove `--schema-only`, `--path-only`, and `--no-upgrade-diff` from lockfile-era `schema` commands.
- Those flags belong to the older clean/upgrade split model and are not part of lockfile/checkpoint A/B flow.

6. Help text consistency.
- Root and panel help must describe the de-duplicated command surface only.
- Remove references to mixed behavior like \"schema diff ... live drift\".

Help Output Contract (v1)
1. Root help (`matey --help`) must expose only canonical command surface.
- Show namespaces: `schema`, `db`, `lock`, `ci`, `cd`, `config`.
- Do not show removed commands/aliases (`up`, `down`, `rollback`, `schema diff --live`, etc.) at top level.
- Summary lines must match actual behavior:
  - `schema`: artifact-only.
  - `db`: live-only.
  - `lock`: lockfile health/maintenance.

2. `schema --help` and subcommand help must match lockfile-era flags.
- `schema validate`: show `--clean`, `--down` only (plus shared globals).
- `schema regen`: show `--clean` only (plus shared globals).
- `schema diff`: no command-local flags (plus shared globals).
- Do not show legacy flags: `--schema-only`, `--path-only`, `--no-upgrade-diff`, `--live`, `--expected`.

3. `db --help` and subcommand help.
- Mutators shown: `up`, `migrate`, `down`.
- `rollback` absent.
- Help descriptions must reflect semantics:
  - `up`: create-if-needed + apply pending migrations.
  - `migrate`: apply pending migrations only (no implicit create).
  - `down [N]`: roll back `N` steps (default 1).
- `db diff` help shows live URL selection behavior; no `--clean`/`--expected`.

4. `lock --help` and subcommand help.
- Show `lock doctor` and `lock sync` only.
- `lock doctor` documented as read-only.
- `lock sync` documented as deterministic rewrite from repo artifacts.

5. No duplicate action surfaces in help.
- `ci init --print` and `cd init --print` must not appear.
- Keep `ci print` and `cd print` as the print path.

Why lockfile is still needed when every step has a checkpoint
1. Binds migration state to checkpoint state cryptographically.
2. Detects reorder/rename/delete/history rewrite deterministically.
3. Provides O(n) divergence detection via chain comparison.
4. Verifies `schema.sql` matches declared head state.
